# Cloudflare Pages migration

## Current deployment/configuration

- Static site generated by Jekyll.
- Production deploy currently runs from `.github/workflows/jekyll-gh-pages.yml`.
- The workflow builds `_site` from the `main` branch and deploys to GitHub Pages.
- The site URL is configured as `https://liquidglassresources.com` in `_config.yml`.
- The repo does not contain a `CNAME` file, which suggests the custom domain is configured in GitHub Pages settings.
- CSS is served from the committed file `assets/dist-styles.css`.
- Algolia and PostHog are client-side integrations that should continue working after hosting changes.

## Target Cloudflare Pages configuration

Configure a Cloudflare Pages project with:

- Production branch: `main`
- Root directory: `/`
- Build command: `jekyll build`
- Output directory: `_site`
- Environment variable: `RUBY_VERSION=3.1.3` on Preview and Production

## Required DNS changes

1. Ensure `liquidglassresources.com` is managed as a Cloudflare zone.
2. If the zone is not already in Cloudflare, update registrar nameservers to Cloudflare.
3. Add the following custom domains to the Cloudflare Pages project:
   - `liquidglassresources.com`
   - `www.liquidglassresources.com`
4. Keep apex as the canonical hostname and redirect `www` to apex.
5. After Cloudflare is serving production, remove the custom domain from GitHub Pages.
6. Remove legacy GitHub Pages DNS records after cutover.
7. Verify no restrictive CAA records prevent certificate issuance.

## Repo preparation changes

- Add `.ruby-version` to pin the Ruby version used by Cloudflare Pages and local development.
- Commit `Gemfile.lock` and stop ignoring it.
- Document the build and deployment process in `README.md`.
- Keep the initial migration Ruby-only; do not couple this with a Tailwind CI rebuild change.

## Risks

- Build drift until `Gemfile.lock` is committed.
- Certificate issuance or DNS propagation delays during cutover.
- Split-brain production behavior if GitHub Pages still owns the custom domain after Cloudflare goes live.
- Preview environments are not fully isolated because some assets reference the production domain.
- CSS changes remain partially manual until a reproducible CSS build step is added.

## Rollout steps

1. Commit repo prep changes.
2. Generate and commit `Gemfile.lock`.
3. Create the Cloudflare Pages project and set `RUBY_VERSION=3.1.3` for Preview and Production.
4. Validate the `pages.dev` preview deployment.
5. Smoke test homepage, posts, subcollection pages, sitemap, assets, search, and analytics.
6. Schedule a short content freeze.
7. Move or confirm DNS in Cloudflare.
8. Attach apex and `www` custom domains in Pages.
9. Wait for HTTPS and domain activation.
10. Remove the custom domain from GitHub Pages.
11. Remove legacy GitHub Pages DNS records.
12. Monitor production for at least 24 hours.
13. Remove or archive the GitHub Pages workflow after the rollback window closes.

## Rollback plan

- Before DNS cutover, keep GitHub Pages as the public production host.
- After DNS cutover, rollback by restoring the GitHub Pages DNS targets and reattaching the custom domain in GitHub Pages.
- If the failure is deployment-related rather than DNS-related, redeploy the last known good Cloudflare deployment while debugging.
- Keep the old GitHub Pages workflow and previous DNS values available until Cloudflare has been stable for at least 24 hours.

## Acceptance criteria

- `https://liquidglassresources.com` is served from Cloudflare Pages with valid HTTPS.
- Canonical redirect behavior is correct between apex and `www`.
- Pushes to `main` trigger successful Cloudflare Pages deployments.
- Homepage, representative posts, and subcollection pages render without missing assets.
- Algolia search works as expected.
- Sitemap and SEO metadata still point at the production domain.
- GitHub Pages no longer serves the production custom domain.
- Rollback steps are documented and can be completed quickly.
